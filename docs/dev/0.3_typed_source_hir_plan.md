# CatSeq 0.3 Typed Source HIR implementation plan

Status: implementation in progress; production-path vertical slice operational

## Goal

Implement one production-path vertical slice that statically compiles a
versioned Compile Request into Python-free, definition-segmented Typed Source
HIR with stable Type Signatures and diagnostics. CatSeq validates the language
and compiler with repository-owned source bundles; production repositories may
apply the same contract to larger experiment entries such as Rydberg Transfer.
The same slice introduces the minimum rustc-style query infrastructure needed
to prove incremental reuse.

This is one acceptance milestone rather than a sequence of temporary compiler
architectures. Its internal components may be built in dependency order, but no
intermediate tree-sitter, live-Python, recursive-object, or non-incremental path
becomes a supported boundary.

## Delivered data path

```text
CompileRequest
  -> nac3parser / nac3ast
  -> static Source Bundle import resolution
  -> Module Index and per-definition projections
  -> Compile Reachability
  -> Compile Class Schemas, Compile Instances, and lazy Global Definitions
  -> per-definition Source HIR Segments
  -> name and call resolution
  -> recursive-SCC type, availability, and dependency-role analysis
  -> Type Signatures, Semantic Facts, summaries, and Diagnostic Sets
  -> CheckResponse with Incremental Stats
```

The binary frontend never imports a project module through CPython. Existing
source spelling for dataclasses, replacement, properties, service/module
singletons, channel dictionaries, and scan lookup is recognized statically and
normalized to the closed 0.3 type universe. Compile-reachable legacy explicit
state-map calls are migrated to the already accepted implicit Morphism Effect
contract rather than preserved as a native source type.

## Query engine required by the slice

The implementation includes these logical query families:

```text
SourceText(ModuleKey)
ParseModule(ModuleKey)
IndexModule(ModuleKey)
DefinitionHeader(DefinitionKey)
LowerSourceHir(DefinitionRevision)
ResolveDefinition(DefinitionRevision)
ReachableDefinitions(EntryKey)
InferTypes(SccKey)
InferAvailability(SccKey)
AnalyzeDependencyRoles(SccKey)
CollectDiagnostics(EntryKey)
```

Query keys use stable semantic identities, including normalized Definition
Revisions where implementation identity is part of the query. Query execution
records ordered reads in a current DepGraph while consulting an immutable
previous DepGraph. The implementation provides stable key and result
fingerprints, `try_mark_green`, selected on-disk result caching, cache
promotion, a private session directory, and atomic successful publication.

Module Indexes, Source HIR Segments, resolved owner facts, Type Signatures, and
analysis summaries are eligible for on-disk result caching. Parsed AST values
need not be persisted initially; unchanged downstream query results can remain
green without loading an AST. Source Anchor-to-span data for an unchanged raw
source input remains available to render cached diagnostics.

All providers obey the pure-provider contract. Diagnostics are returned as
stable data. Arena-backed owner stores, if used for Source HIR, publish their
segments only through the query engine after successful construction.

## Required source semantics

The slice supports the source constructs exercised by the reachable Rydberg
Transfer graph:

- absolute and relative static imports over the Source Bundle;
- declaration indexing without module execution;
- explicit Compile Entry and transitive method/property reachability;
- registered dataclass and dataclass-transform class schemas;
- lazy pure module globals and immutable Compile Instances;
- the closed scalar, native-handle, sequencing, and contextual type universe;
- lexical names, locals, fields, calls, attributes, fixed aggregates, and
  registered Intrinsic operations;
- Compile-only structural `if` and native sequencing `for` recognition;
- typed scan lookup as Link values without link values selecting topology;
- stable Source Anchors for every diagnostic emitted by the slice.

Unsupported reachable Python behavior is a compile diagnostic. Unreachable Host
imports, methods, lifecycle fields, and module-level setup do not fail the
compile.

## Explicitly outside this slice

This milestone does not emit the Value Expression arena, Morphism arena,
Morphism Effects, Canonical Program, RTMQ fragments, Relocatable Artifact, or
OASM Call Plan. It does not retain the current tree-sitter HIR or pinned Python
payloads as a compatibility path. Hardware conditionals and general Python
object mutation remain outside 0.3.

The next native abstract-evaluation work consumes these exact Typed Source HIR
segments and query summaries; it does not introduce a second frontend.

## Consumer acceptance contract

Production consumers can apply the following checks to their own Source Bundle.
These checks run in the consumer repository against an installed CatSeq build;
CatSeq does not fetch, import, or depend on the consumer repository:

1. `catseqc check` resolves every compile-reachable experiment, service, and
   module definition without starting CPython or importing the application
   package.
2. Every reachable definition receives a stable Type Signature, and supported
   expressions receive type, availability, resolution, and dependency-role
   facts.
3. A no-change second process marks module, definition-owner, and SCC analysis
   queries green and loads only root results required for the response.
4. Changing comments or whitespace reruns source parsing but stops at an
   unchanged semantic fingerprint.
5. Changing an unreachable Host-only method does not invalidate reachable Typed
   Source HIR or Type Signatures.
6. Changing one compile-visible field, definition body, or signature makes only
   its actual owner/SCC and semantic dependents red.
7. A reachable Host call, unsupported dynamic import, Link-controlled topology,
   or incompatible type produces a deterministic Source-Anchor diagnostic.
8. Failed compilation leaves the previous incremental session usable.
9. Cold and warm wall time, per-DepKind executions, green/red counts, result
   cache loads, bytes read/written, and fingerprint time are reported.

Tests include focused query-engine and type-system units, synthetic multi-module
incremental cases, and the Rydberg Transfer integration fixture. Repeated runs
must verify both semantic outputs and the expected red/green query boundary;
timing alone is not proof of reuse.

## Completion handoff

At completion, the implementation report records the measured cold and warm
profiles, remaining executed queries on a no-change run, unsupported reachable
source constructs, and the exact Typed Source HIR root for the Rydberg Transfer
entry. Only then does implementation proceed to native abstract evaluation and
Morphism arena emission.

## Implementation checkpoint (2026-07-14)

The first production-path vertical slice is now executable. Both text and JSON
forms of `catseqc check` use the revision-pinned NAC3 parser and CatSeq-owned
Typed Source HIR; the CLI no longer selects the older tree-sitter/arena path.
The source loader resolves modules lazily from static import names and never
imports the application through CPython.

`catseqc check --format json` is a compact schema-versioned response containing
the entry signature, definition/node counts, diagnostics, and incremental
statistics. It deliberately excludes HIR. Full definition-owned HIR is an
explicit debug projection through `catseqc emit-hir --format json`, written
directly to a buffered serializer without constructing an intermediate
`serde_json::Value` tree.

For `RydbergTransferExp.build_sequence`, the current frontend reaches 47 source
definitions in 22 modules and publishes 4,176 flat HIR nodes. All 665 reachable
call nodes have a resolved definition and a result type. Legacy `seq`,
`get_end_state`, and explicit state-map edges are absent from the published HIR.
Static properties and comprehensions expand the known module instances used by
`InitializeService.init`. Definitions implemented through `oasm_black_box` are
recognized generically as opaque atomic Morphism boundaries, without lowering
their legacy Python/OASM bodies or naming rb1-specific methods in the Intrinsic
Registry. Production repositories own their integration checks against this
frontend; the standalone CatSeq suite uses only CatSeq-owned source fixtures.

The persistent query session stores flat nodes and edge ranges with separate
key and result fingerprints. Summary metadata and the full HIR report are
separate cache payloads. A cold check executes 329 queries. The immediately
following process marks all 329 nodes green, loads one cached summary, runs zero
providers, and writes zero bytes. It reads 78,754 bytes; `emit-hir` additionally
loads the full payload for a total of 1,544,153 bytes. Comment-only edits stop
after one parse fingerprint; a reachable service-body or compile-field edit
executes one `LowerSourceHir`; a failed check does not replace the previous
successful session. Compiler, cache-format, and Intrinsic Registry semantic
versions are all part of cache compatibility.

Release measurements on this workspace, including process startup and static
source loading, are shown separately for the compact response and the explicit
1.47 MB JSON HIR projection:

| Output / case | Repetitions | Total | Mean | Peak RSS |
| --- | ---: | ---: | ---: | ---: |
| Compact JSON check, cold unique cache | 20 | 1.41 s | 70.5 ms | about 13.9 MB |
| Compact text check, warm unchanged cache | 100 | 0.22 s | 2.2 ms | about 3.6 MB |
| Compact JSON check, warm unchanged cache | 100 | 0.26 s | 2.6 ms | about 3.6 MB |
| Full HIR JSON, cold unique cache | 20 | 1.40 s | 70.0 ms | about 13.8 MB |
| Full HIR JSON, warm unchanged cache | 100 | 0.75 s | 7.5 ms | about 6.7 MB |

The cold session serializes 1,544,153 bytes across the dependency graph/summary
and full HIR payload. Stable fingerprint work accounts for about 6.7 ms of the
cold run. The normal warm path is process startup plus decoding the 78 KB
summary cache. The full JSON path additionally loads and streams every HIR node
and semantic fact, but no longer materializes a second JSON object graph. Kernel
policy in the development container disallows hardware-counter and `ptrace` CPU
sampling, so these hotspot statements are based on phase/query counters,
response size, and controlled wall time rather than a `perf` call graph.

Source-anchored errors now cover unsupported reachable statements, Host calls,
Link-controlled structure, and known return-type mismatches. The existing 185
Python tests pass unchanged, as do the complete Rust workspace and strict
Clippy checks.

This checkpoint is not the completion handoff. The next frontend work still
needs a real recursive-SCC constraint solver, owner/projection result values
instead of analysis facades over one lowering pass, structured cached
`DiagnosticSet` values, formal-parameter checking at resolved call sites, a
versioned serialized `CompileRequest`, and fuller static Compile Class/global
schemas.

The subsequent first abstract-evaluation slice is now implemented:
`catseqc emit-arena` emits a Python-free, definition-linked canonical Morphism
arena for the same real entry. It has variadic Serial/Parallel nodes,
channel-bound Instantiate leaves, DefinitionRef leaves, flat edge/boundary
tables, and no Source HIR owner. Scalar/link operands and call arguments still
need the separate Value Expression arena before the artifact is executable.
