# CatSeq 0.3.1 Linux raw-Ethernet runtime migration plan

Status: Phases 0–5 complete; Phase 6 live chassis-2 acceptance passed on
2026-07-23, with its exact-commit checkpoint still pending

Baseline: `feat/0.3.1-oasm-plan` at `fd0f64c`

Archived implementation source: `feat/0.4-rust-rtmq-backend` at `7c9f02d`

This document is the execution checklist for the 0.3.1 runtime goal. A phase is
complete only after its gate passes. Checkboxes record verified work, not work
that merely has code written for it.

## Goal

Close the in-memory path from the Rust-produced `OASMCallPlan` to physical RTMQ
execution while keeping OASM as the final instruction encoder and moving all
network I/O, Download-loader materialization, RTLink framing, retry policy, and
completion monitoring into Rust.

The only production transport in 0.3.1 is Linux raw Ethernet using
`AF_PACKET/SOCK_RAW`. The production path must not use pcap and must never call
OASM `run_cfg`, `assembler.run()`, or `sequence.run()`.

The completed path is:

```text
restricted Python source
  -> Rust compiler
  -> OASMCallPlan
  -> decoded OASM calls
  -> assemble_oasm_calls()                    Python/OASM, no I/O
  -> AssembledOASMProgram                     Rust-owned frozen PyO3 value
  -> execute_oasm_program()                   Python facade
  -> catseq._native.execute_oasm_program()    direct PyO3 handoff, GIL released
  -> catseq-runtime
  -> materialize_download_loader()
  -> RTLink frames
  -> Linux RawEthernetTransport
  -> terminal completion or structured failure
```

ADR 0003 remains unchanged in meaning: native **compilation** ends at the
`OASMCallPlan`. `AssembledOASMProgram` is a post-compiler runtime handoff, not a
new compiler artifact and not a Rust replacement for OASM encoding.

## Fixed scope and decisions

1. OASM remains the final encoder for board ICH instructions.
2. Rust owns Download-loader generation, RTLink framing, raw Ethernet, and
   runtime monitoring.
3. The handoff is in memory. Saving or loading an artifact is optional future
   work and is not required by any 0.3.1 interface.
4. `assemble_oasm_calls` is the assembly interface. It does not execute a
   sequence and does not return a runnable OASM sequence.
5. `execute_oasm_program` is the execution interface. It never performs source
   compilation or OASM assembly.
6. Linux raw Ethernet is the only production transport. No pcap, FT601, UDP,
   Windows, or macOS transport is added.
7. The `Transport` seam is retained because it has two real adapters:
   `RawEthernetTransport` for production and `InMemoryTransport` for runtime
   state-machine tests.
8. The archived runtime is migrated module by module. Commit `7c9f02d` is not
   cherry-picked as a whole because its interface is coupled to the abandoned
   `NativeProgram`, `DeploymentBindings`, target fingerprints,
   `catseq-rtmq-ir`, and native OASM backend.
9. No implicit mock execution or fallback is allowed. Missing OASM, invalid
   topology, insufficient raw-socket permissions, and protocol failures are
   explicit failures.
10. Consumer-specific experiment lifecycle integration and a one-call
    `compile_and_run` convenience interface are follow-up work. This goal
    implements and verifies every module method needed by that later wiring.

## Runtime handoff interfaces

### Assembled program

The Python/OASM adapter returns a Rust-owned frozen PyO3 value with this
logical schema. There is no duplicate Python dataclass model:

```text
AssembledOASMProgram {
    schema_version: 1,
    reply_node: u16,
    reply_channel: u8,
    boards: [
        AssembledOASMBoard {
            address: "main" | "rwg0" | ...,
            ich_words: [u32, ...],
            exception_handler_word: u32,
        },
        ...
    ]
}
```

The interface invariants are:

- at least one board is present in an executable program;
- board addresses are unique and use the existing `OASMAddress` spelling;
- `reply_node` and `reply_channel` are copied from the OASM encoding context
  because OASM embeds that host endpoint in the completion/exception epilogue;
- `ich_words` are the finalized ICH program, not OASM's outer Download loader;
- `exception_handler_word` indexes the same `ich_words` array;
- the value contains no interface name, MAC address, physical **board** node,
  timeout, loader words, socket, or OASM object;
- the PyO3 value owns the Rust schema directly; JSON, a byte request, and a
  persistence format are not required at this boundary.

### Runtime configuration

The runtime accepts one Rust-owned frozen PyO3 Linux configuration value with
this logical schema:

```text
LinuxRawEthernetRuntimeConfig {
    schema_version: 1,
    interface: str,
    destination_mac: optional [u8; 6],
    timeout_ms: u64,
    boards: [
        BoardEndpoint {
            address: OASMAddress,
            node: u16,
            channel: u8,
            instruction_capacity_words: usize,
        },
        ...
    ]
}
```

The raw transport derives the source MAC from `interface`. When
`destination_mac` is absent it preserves pinned OASM behavior by using source
MAC plus two, with overflow rejected during preflight. The production protocol
constants remain implementation details: EtherType `0xFACE`, 32 Ethernet
padding bytes, Download flag `4`, Download tag `0`, and 14-byte RTLink frames.
Callers do not configure them.

The board map must contain every and only assembled board address, physical
nodes must be unique, every endpoint channel and the assembled reply channel
must fit the pinned OASM protocol, every program must fit its physical ICH
capacity, and every exception handler must index its own program. Download
frames use each physical board endpoint's channel. Incoming frames must target
the reply node/channel already encoded in the assembled program; the runtime
does not accept a second potentially contradictory copy. All preflight checks
finish before the raw socket opens.

The archived runtime used one `host_channel` for both directions. That coupling
must not be migrated: pinned OASM `run_cfg.chn` selects the target channel for a
Download frame, while `intf.loc_chn` is encoded as the host reply channel. Tests
must include unequal values so the two roles cannot collapse again.

### Runtime result

Success and failure retain the archived evidence model:

- per-board monotonic states: `not_dispatched`, `loader_submitted`,
  `launch_submitted`, `succeeded`, and `device_failed`;
- execution certainty: `not_started`, `fully_observed`, or `indeterminate`;
- device exception flags and optional instruction address;
- per-board returned data;
- stable failure codes separate from diagnostic text.

The Python facade raises a typed `CatSeqRuntimeError` for a failed result while
preserving the complete structured failure on the exception. PyO3 or schema
failures must not be mistaken for device failures.

## Module ownership

```text
catseq/compilation
  decode OASMCallPlan, invoke the supplied OASM assembler, finalize ICH images,
  construct the native PyO3 handoff values, decode the native runtime result

rust/catseq-python
  frozen PyO3 wrappers over the Rust runtime models and a thin execute adapter;
  release the GIL around runtime execution

rust/catseq-runtime
  own the only assembled-program and runtime-configuration schemas, validate
  topology, materialize loaders, dispatch RTLink, own transport policy, monitor
  completion, return structured evidence

rust/catseq-rtmq
  RTMQ instruction codec and OASM-compatible Download-loader materialization

OASM
  encode board instructions and final ICH completion/exception epilogue only
```

`catseq-runtime` is the deep runtime module. Its external interface is one
execute operation over an assembled program and a runtime configuration. Raw
socket, framing, retry, monitoring, and evidence transitions remain inside the
module. The generic `Transport` seam is internal to the Rust implementation and
its tests; it is not exposed as a Python plugin interface.

## Archived implementation reuse matrix

| Archived path at `7c9f02d` | Migration decision | Required change |
| --- | --- | --- |
| `catseq-runtime/src/protocol/rtlink.rs` | Reuse | Remove no behavior; retain exact encode/decode and odd-word zero padding. |
| `catseq-runtime/src/transport/mod.rs` | Reuse | Keep `Transport`, send certainty, and `InMemoryTransport`; keep it internal to the runtime module. |
| `catseq-runtime/src/transport/raw_socket.rs` | Reuse with one fix | Keep `AF_PACKET`, `poll`, `recv`, and frame layout; derive source MAC at open and hide fixed protocol constants. Use `sendto` with an explicit `sockaddr_ll` carrying the interface index and chassis MAC because bound `send` did not put frames on the wire on the validated host. |
| `catseq-runtime/src/execution.rs` | Adapt state machine | Replace `NativeProgram` and `BoardId` with `AssembledOASMProgram` and `OASMAddress`; keep evidence, retry, monitoring, and contradiction handling while separating target and reply channels. |
| `catseq-runtime/src/topology.rs` | Rewrite interface | Remove `BoardKind`, fingerprint, and native-program validation; validate address-to-node/capacity bindings. |
| `catseq-runtime/tests/rtlink_parity.rs` | Adapt fixture setup | Feed the frozen OASM ICH image and exception entry through the current Rust loader. |
| `catseq-runtime/tests/runtime.rs` | Adapt fixtures | Construct assembled OASM programs directly; retain every applicable fault and evidence case. |

The migration must not restore archived `board_program`, relocation, native
instruction compilation, `NativeProgram` serialization, or
`catseq-rtmq-ir` dependencies.

## Explicitly outside 0.3.1

- migrating the complete OASM Python package or its `dev.*` facades into
  CatSeq;
- implementing OASM instruction encoding, `asm(word)`, registers, ports, or
  device helpers in Rust;
- restoring the archived full-native OASM/backend commits after `473c41d`;
- `NativeProgram`, relocation, target fingerprints, or a persistent native
  artifact format;
- required explicit save/load operations;
- a `catseqc run`/`download` command or application-level one-call runner;
- pcap, `pcapy-ng`, FT601, UDP, Windows raw Ethernet, macOS/BPF, or transport
  plugin discovery;
- OASM `run_cfg`, OASM monitor threads, `assembler.run()`, or `sequence.run()`;
- silent mock execution or success without physical terminal evidence;
- application-specific experiment lifecycle wiring or topology discovery;
- changing the restricted source language or the `OASMCallPlan` compiler
  schema for runtime convenience.

## Definition of done

The goal is complete only when all of the following are true:

1. `compile_entry(...).to_oasm_calls()` can be assembled into an immutable,
   versioned in-memory program containing exact per-board ICH words and
   exception entries.
2. The same program can be handed directly to PyO3 without saving it.
3. Rust validates topology before side effects, materializes the pinned-OASM
   compatible Download loader, emits exact RTLink frames, and uses Linux raw
   Ethernet without pcap.
4. Rust returns success only after every board reaches a trusted terminal
   completion; every failure preserves conservative execution evidence.
5. Offline OASM parity, Rust state-machine tests, Python/PyO3 tests, strict
   linting, and package smoke tests pass.
6. Authorized hardware validation passes, or the implementation is reported as
   offline-complete but hardware-unverified rather than being called complete.
7. No supported interface or documentation instructs callers to invoke
   `sequence.run()` or the old `execute_oasm_calls` mock path.
