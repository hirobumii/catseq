# CatSeq 0.3 native compiler

CatSeq 0.3 compiles a restricted Python sequencing language into a Rust-owned,
append-only DAG. The source spelling of `Morphism`, `MorphismDef`, `>>`, `@`,
`|`, and channel dictionaries remains unchanged; the production compiler does
not import or execute the experiment module.

## Compiler ownership

`catseqc` owns the production compilation request. It reads Python source,
discovers explicitly marked sequence entries, resolves their reachable service
templates, and applies CatSeq semantics only to that reachable graph. Host-side
experiment setup, analysis, device I/O, reflection, and arbitrary Python calls
are outside the compiled language even when they share the same source file.

The crate boundary is:

```text
catseqc
  -> catseq-frontend  source loading, parsing, names, restricted semantics
  -> catseq-core      definitions, arenas, passes, specialization cache
  -> catseq-rtmq      relative RTMQ artifacts, scan relocation, board linking
  -> OASM
```

The optional `catseq-python` PyO3 crate is a compatibility and diagnostic
adapter. It is not the production compiler entry and is not required to parse a
sequence. A future long-lived service may call the same frontend/core crates,
but spawning a process for every scan point is not part of the design.

This organization follows the useful frontend/core/target separation in NAC3,
while deliberately excluding its LLVM/inkwell code generator and general
Python runtime model. CatSeq's target backend is RTMQ.

The initial parser uses tree-sitter-python and produces CatSeq-owned HIR. NAC3
is an architectural reference, not a source dependency: this keeps the frontend
independent from NAC3's LLVM-coupled value/code-generation interfaces.

## Restricted Python boundary

The parser may read a normal Python module, but semantic lowering starts only
at `@arena_build` entries and follows known sequence/service calls. The initial
language admits:

- local assignments, return, numeric and boolean constants;
- attribute lookup on declared experiment/service/module values;
- calls to compiler intrinsics and registered template definitions;
- `>>`, `@`, `|`, channel dictionaries, and supported arithmetic expressions;
- structural `if` and bounded `for` when their controlling values are known at
  specialization time.

Scan values are symbolic. They may feed supported timing and operation
expressions, but cannot select imports, change loop trip counts, invoke Python,
or otherwise change DAG topology. Unsupported calls are compile errors rather
than fallbacks to the Python interpreter.

Scan containers are identified by their `ExpParams` annotation, not by a magic
parameter name. Parameters follow normal lexical shadowing rules over module
imports. Scan slots and call targets are collected only from HIR nodes reachable
from the function's returned root. A scan-derived callable or channel-dictionary
key is rejected because it would change DAG topology.

For `RydbergTransferExp.build_sequence`, this means:

- `params[self.pulse_time]` resolves to a runtime scan slot;
- `self.amp_420`, `self.amp_1013`, and `np.pi` resolve as compile-time values;
- service method calls resolve to stable template definitions;
- `get_end_state` is a compiler intrinsic evaluated by a state-analysis pass;
- unrelated methods such as `prepare_run` are not compiled.

## Current vertical slice

`catseqc check <file> --entry <Class.method>` currently performs source parsing,
entry discovery, restricted statement/expression lowering, import-path
resolution, scan-slot allocation, and scan-topology validation. It retains the
parsed syntax tree so lowering does not parse each selected function again. It
rejects unsupported statements in the selected sequence instead of executing
them through Python. The real `RydbergTransferExp.build_sequence` lowers to
source HIR without importing the experiment module.

The following work is intentionally still pending:

- resolve imported service instances to their defining classes and methods;
- recursively compile reachable service/module functions to `DefinitionId`s;
- evaluate compile-time class constants and structural control flow;
- lower resolved source HIR into the shared Morphism arena;
- generate complete RTMQ operation relocations, multi-board linking, and OASM
  calls.

## Storage model

`ArenaStore` is the single logical store. Physical segments separate module
templates, service-bound templates, and experiment programs without requiring
node copies. Nodes use store-checked `(segment, local node)` references. Mutable
segments are append-only; published template segments are immutable.

Template instantiation appends one `Instantiate` node. The compiler traverses
the referenced template with an explicit stack and an instance environment; it
does not expand the template into the program segment. A frozen program pins the
store and root. Later parents cannot change the graph reachable from that root.

## Required invariants

- A node can only refer to nodes in the same logical store.
- Template segments cannot depend on program segments.
- Publishing a template freezes its segment.
- Composition never mutates either input root.
- Freezing never copies or consolidates reachable nodes.
- Compiler traversal is iterative and carries template instance context.
- Python objects and Python callables do not become native DAG payloads.

## Compilation artifacts

Module templates retain channel and parameter slots. A service artifact binds
hardware maps, calibration snapshots, and structural parameters while keeping a
relative event fragment. Runtime binding supplies scan parameters and performs
incremental relocation and board linking. Absolute OASM calls are produced only
after linking.

Stable `DefinitionId` values index an append-only definition table. A template
specialization key contains its definition, structural parameter digest,
hardware-map digest, and calibration digest. It intentionally excludes scan
values. Changing a scan point therefore evaluates runtime slots and relinks the
same RTMQ artifact instead of reparsing source or recompiling the template.

The Python DAG compiler remains the correctness oracle during 0.3 development.
`catseqc` becomes the default only after differential Rydberg tests match
events, calls, and integer assembly.
