# CatSeq 0.3 native compiler

CatSeq 0.3 compiles a restricted Python sequencing language into a Rust-owned,
append-only DAG. The source spelling of `Morphism`, `MorphismDef`, `>>`, `@`,
`|`, and channel dictionaries remains unchanged; the production compiler does
not import or execute the experiment module.

## Compiler ownership

`catseqc` owns the production compilation request. It reads Python source,
discovers explicitly marked sequence entries, resolves their reachable service
templates, and applies CatSeq semantics only to that reachable graph. Host-side
experiment setup, analysis, device I/O, reflection, and arbitrary Python calls
are outside the compiled language even when they share the same source file.
Compile Reachability applies to individual definitions rather than declaring an
entire class or module compiled.

The same source class may therefore have two independent consumers. CPython
uses a Host Object for setup, persistence, analysis, and device lifecycle. The
binary compiler creates an immutable Compile Instance with a stable
`InstanceId` and complete projection of compile-reachable fields from source
declarations and Compile Environment bindings. It does not import the Host
Object or execute its metaclass, generated initializer, or `__post_init__`. A
field written only by host lifecycle code is unavailable to compile-reachable
methods.

The crate boundary is:

```text
catseqc
  -> catseq-frontend  source loading, parsing, names, restricted semantics
  -> catseq-core      definitions, arenas, passes, specialization cache
  -> catseq-rtmq      relative RTMQ artifacts, scan relocation, board linking
  -> OASM Call Plan
  -> Python OASM assembler adapter
```

The platform wheel packages `catseqc` beside the Python sources. The public
`compile_entry()` facade invokes that compiler without exposing its process
transport to the host application. There is no Python-to-Rust arena adapter or
second Python compiler entry. `catseqc` follows rustc's one-shot model: it loads
a previous on-disk incremental session, applies red-green query validation,
atomically publishes the successful replacement session, and exits. A resident
worker may call the same crates as an optional deployment optimization but is
not required for correctness or cache reuse.

The Rust-owned 0.3 compiler ends at an `OASM Call Plan`: deterministic,
epoch-segmented and board-grouped target calls after RTMQ linking. Calls within
an Epoch use offsets from its initial or Sync Phi release origin. The existing
Python OASM assembler encodes that plan. Direct Rust generation of RTMQ binaries
is outside 0.3; the adapter must not perform scheduling, state analysis, scan
evaluation, or any other CatSeq compilation semantics.

The production API is a versioned one-shot Compile Request. It names a Source
Bundle manifest, canonical entry, Compile Environment, Target Profile,
incremental-cache location, and optionally a batch of Link Bindings. The
response contains an artifact key, Link Schema, Python-free Relocatable
Artifact, optional OASM Call Plans, diagnostics, and incremental reuse
statistics. No request or response contains a live Python object or AST.

This organization follows the useful frontend/core/target separation in NAC3,
while deliberately excluding its LLVM/inkwell code generator and general
Python runtime model. CatSeq's target backend is RTMQ.

The production syntax layer uses revision-pinned `nac3parser` and `nac3ast`
crates, which do not depend on LLVM. Their structured Python AST is normalized
immediately into CatSeq-owned Source HIR. The current tree-sitter vertical slice
is migrated and then removed rather than retained as a second syntax frontend.
The local NAC3 checkout is a persistent reference, not a path dependency, and
no NAC3 core type or code-generation interface crosses the Source HIR boundary.

Explicit dataclasses and registered `typing.dataclass_transform` class families
produce Compile Class Schemas statically. Fields, class constants, methods, and
properties are indexed without executing a decorator or metaclass. Only fields
actually reached by compiled definitions are materialized into a Compile
Instance, so unsupported host-only defaults and factories remain unevaluated.

## Type-analysis boundary

Parsing annotations is not type analysis. CatSeq's first 0.3 frontend milestone
is a Rust-owned semantic pipeline:

```text
source HIR
  -> reachable definition collection
  -> name resolution
  -> type-constraint generation
  -> type unification
  -> Typed Source HIR
  -> native abstract evaluation and Python-syntax normalization
  -> native Value Expression and Morphism arenas
```

Annotations contribute constraints, but existing service code is not required
to annotate every local, field, parameter, or result. The compiler may infer
those types from restricted constructor assignments and reachable call
relationships. Unresolved ambiguity is a compile error, not a request to import
or execute Python.

The syntax crates are reused directly, but the type checker is CatSeq-owned.
It follows NAC3's proven separation of top-level definition registration,
symbol resolution, constraint generation, union-find unification, and error
aggregation without depending on `nac3core`. NAC3's `TopLevelComposer`,
`SymbolResolver`, inferencer, and unifier are implementation references rather
than runtime types or crate interfaces. CatSeq constraints operate on CatSeq
`DefinitionId`s and its deliberately smaller type set, not NAC3's LLVM-,
NumPy-, or ARTIQ-oriented type universe.

The compiler session stores Source HIR in one logical arena segmented by
`DefinitionId`. Nodes and child edges occupy flat ranges, and cross-definition
calls store definition identities instead of pointers into another segment.
Resolution, type, Availability, and compile-evaluation results are semantic
side tables keyed by HIR node identity. Typed Source HIR is this immutable
segment plus complete facts; it is not a cloned typed tree. Replacing one
definition revision therefore does not copy unchanged module or service HIR.

The Compile Environment complements this analysis with source-external facts:
hardware mappings, calibration snapshots, structural instance data, and
registered intrinsic signatures. It does not replace source-level type
inference and does not contain runtime scan values.

Service and module instances are immutable Compile-time `InstanceId`s. Method
calls resolve statically to an instance and `DefinitionId`; instance fields are
resolved before arena lowering, and compiled source cannot mutate `self` or a
nested Python object. Sequence-dependent logical fields such as
`MWModule._tracker.phase` resolve to a registered `PhaseFrameId` instead. The
Module declares a reusable `PhaseFrameDef`; the Compile Environment binds its
semantic roles to physical channels and calibrated offsets for one instance.
Typed Source HIR resolves the resulting ID; Morphism Effects store the ID and
native Value Expressions; Serial threads the frame environment. The compiler
does not infer a frame from coincident channel writes. Physical per-channel
oscillator phase remains a separate hardware/Atomic Operation concern.

The confirmed Morphism-family source types currently include:

- `Morphism`, a channel-bound sequencing state transformer;
- `MorphismTemplate`, spelled `MorphismDef` by the existing Python API, with
  free channel slots;
- `AtomicOp`, a sealed family whose hardware-event variant represents existing
  `AtomicMorphism` values and whose opaque-region variant represents existing
  `TimedRegion` and `BlackBoxAtomicMorphism` values.

The confirmed scalar base types are `Unit`, `Bool`, `Int64`, `Float64`,
`Duration`, and Compile-only `String`. Python `None`, `int`, and `float` map to
`Unit`, `Int64`, and `Float64` respectively. Target ABI widths such as `u16` or
`u32` are constraints and encodings in Atomic Operation parameter schemas, not
additional source types. Integer overflow is a compile error; restricted source
does not inherit Python's arbitrary-precision integer semantics.

Native handle types are Board, nominal `Channel<ChannelKindId>`, immutable
`Instance<CompileClassSchema>`, and `ScanParam<T>`. Sequencing types are
Morphism, MorphismTemplate, and AtomicOp. Contextual types are Compile-time
Optional, fixed Tuple, registered native Record, Channel Bindings, and Scan
Bindings; no general mutable or dynamically indexed container type exists.

Every source Definition Revision has one Type Signature. CatSeq 0.3 has no
user-defined generic functions and does not monomorphize source definitions by
base type or Value Availability. Native operators, replacement, and Atomic
Schemas may be overloaded in the Intrinsic Registry, but type inference resolves
each use to one concrete intrinsic identity before specialization. Morphism
Template bindings are value parameters rather than source type parameters.

Int64 promotes implicitly only to Float64. Bool does not promote to Int64, and
numeric values never convert implicitly to Duration. `T | None` is accepted only
as a Compile-discriminated Optional. A native Loop induction variable is
`Int64 @ Device` and is accepted only by target-declared loop-register operand
positions; it cannot select topology.

Binding a Morphism Template produces an `Instantiate` leaf of type `Morphism`.
Template composition uses the same variadic Serial and Parallel representation
and never executes a stored Python generator.

Restricted source marks ordinary composite definitions with
`@morphism_template`; their annotated result remains `MorphismDef` for API
compatibility. Hardware-driver leaves may use `@atomic_morphism(stable_symbol)`,
but a symbol becomes compilable only when its Atomic Schema is present in the
versioned Rust Intrinsic Registry. This follows the useful `@kernel` versus
syscall separation in ARTIQ coredevice drivers without adopting ARTIQ's live
Python object embedding: CatSeq resolves source statically and stores only
native Definition, Template, Value, and Channel IDs.

Composite intrinsic APIs are precompiled template bodies. For example,
`linear_ramp` publishes `Serial(load(ramp_waveforms), play, Wait(duration),
load(endpoint_waveforms), play)` and invocation creates an `Instantiate`.
`set_state` is likewise the composite `Serial(load(static_waveforms), play)`;
neither operation is an opaque Atomic Schema. All three parameter lists have
the same `list[WaveformParams]` value type and feed the single RWG `load`
Atomic Schema. Their different calculations are Value DAG dependencies on the
implicit incoming State Environment, not different load operations. `load` is
a time-flexible preload while `play` is an exact-timestamp event. Their
load-to-play deadline relationship survives as a native scheduling dependency
when the RTMQ call plan is emitted.

A Morphism's incoming state is not a source-language parameter. Its Morphism
Effect transforms the implicit State Environment threaded from child to child by
Serial composition. Native source has no `StateMap` type or `get_end_state`
intrinsic; existing services that pass explicit state maps are migrated to make
that dependency structural.

An opaque Atomic Operation has either an `Exact(DurationExpr)` or `Dynamic` Timing
Contract. Exact regions remain within one Epoch. Dynamic regions make the
current exit time unknown and require a Sync Phi before ordinary static
scheduling resumes.

Concrete Durations are non-negative integer Cycle Counts. Symbolic
`DurationExpr` values, including Runtime Slots and Parallel maxima, evaluate to
integer cycles before RTMQ linking. A Logical Timestamp is `(EpochId,
offset_cycles)` and is distinct from a Duration; timestamp arithmetic cannot
cross a Sync Phi boundary.

`Duration` is the only dedicated physical-quantity base type in 0.3.
Frequency, phase/angle, and amplitude use `Float64`; the receiving Atomic
Operation's typed parameter schema supplies their unit convention, range,
permitted Value Availability, and target encoding. A general physical-dimension
type system is outside the 0.3 milestone.

Phase Frames use `Float64` Phase Turns: `0.0` maps to zero and `1.0` maps to
`2π`. Native frame updates use Euclidean modulo one, including Link-time Value
Expressions. Rotation angles in radians remain a distinct Atomic Operation
parameter role. Hardware-specific phase words are quantized only during target
lowering.

Unit conversion and scan binding never round implicitly. A non-integral cycle
quantity is rejected unless its source expression or scan schema explicitly
selects floor, ceiling, or rounding Cycle Quantization. Decimal unit literals
are interpreted exactly before integrality is checked.

Hardware state is internal to Morphism Effects. Each accepted state definition
has a nominal `StateTypeId` associated with an extensible `ChannelKindId`; the
compiler does not hard-code a closed Rust enum of TTL, RWG, and RSP states.
State Types and their fields participate in atomic transition checking but are
not values passed explicitly between services.

Every typed value also carries Compile, Link, or Device Value Availability.
Runtime and Environment Slots produce Link values; native Loop induction
variables and other target-declared operands may produce Device values.
Availability qualifies a base type instead of creating
`Runtime[T]` wrapper types. A separate Dependency Role distinguishes Structural
uses from Relocatable uses. Topology requires Compile values, supported RTMQ
operands may be Link values, and neither Link nor Device values may control a
source `if` in 0.3. Dynamic opaque timing must reconverge through Sync Phi.

These dimensions are computed by three distinct incremental analyses. SCC type
unification publishes Type Signatures; forward lattice data flow publishes
Availability Transfers; use-site and call-edge analysis publishes Structural
Dependency Summaries. Dependency Role is non-exclusive at the value level: one
Compile value may control structure at one use and feed a relocatable atomic
operand at another. It then affects the Specialization Key and may also produce
a relocation, whose binding is checked against the captured specialization.
Type analysis is not rerun per specialization.

The Morphism arena has no container types or container nodes. Typed Source HIR
may temporarily hold Contextual Aggregates: Compile-time optionals, channel
bindings spelled with the existing dictionary API, and fixed aggregates
accepted by an Atomic Operation parameter schema. Specialization and typed
lowering must consume them before arena construction. General mutable
containers and dynamic container indexing are outside the restricted language.

For source compatibility, 0.3 also recognizes construction and
`dataclasses.replace(...)` for registered nominal dataclass schemas. Rust
eagerly normalizes replacement into a complete immutable field vector; it
never imports or executes the Python dataclass, and no lazy update overlay
survives type analysis. Atomic Operation lowering flattens the resulting fields
into constants and Link relocations, so no record survives in the Morphism
arena. Arbitrary dataclass behavior is unsupported.

The Morphism arena is Python-free. Class attributes, calls, replacement,
conditionals, and source containers have already become native constants,
handles, `DefinitionId`s, `ValueExprId`s, or CatSeq nodes when it is built.
Symbolic scan arithmetic lives in a separate native Value Expression Arena;
Atomic Operation payloads refer to it rather than to Source HIR expressions.
Published artifacts own these two native arenas and native provenance/schema
tables only. Source HIR is retained independently by the compiler session for
incremental invalidation and diagnostics and is not pinned by an artifact.

## Restricted Python boundary

The parser may read a normal Python module, but semantic lowering starts only
at `@arena_build` entries and follows known sequence/service calls. The initial
language admits:

- local assignments, return, numeric and boolean constants;
- attribute lookup on declared experiment/service/module values;
- calls to compiler intrinsics and registered template definitions;
- `>>`, `@`, `|`, channel dictionaries, and supported arithmetic expressions;
- structural `if` when its controlling value is known at specialization time;
- bounded sequencing `for` as a native Loop Region, and pure Compile data loops.

Scan values are symbolic. They may feed supported timing and operation
expressions, but cannot select imports, change loop trip counts, invoke Python,
or otherwise change DAG topology. Unsupported calls are compile errors rather
than fallbacks to the Python interpreter. An `if` predicate with Link or Device
availability is likewise a compile error; 0.3 has no hardware Branch node.

### Static import resolution

`catseqc` receives a Source Bundle with explicit project source roots and never
delegates import resolution to CPython, `PYTHONPATH`, import hooks, or live
`PyModule` objects. It classifies modules as:

- Source Modules, whose Python source is parsed statically;
- Intrinsic Modules, whose symbols, types, and lowering rules are registered by
  the Rust compiler; and
- Host Modules, which may remain unloaded when unreachable but are rejected if
  a reachable definition uses them.

Absolute and relative imports and explicit aliases are supported. Wildcard and
dynamic imports and runtime `sys.path` changes are not. Package `__init__.py`
files are never executed; their static re-exports are read only when required.
CatSeq and supported numerical or dataclass operations are preferably supplied
as Intrinsic Modules rather than discovered by interpreting their Python
implementations.

Each Intrinsic Module is supplied by a versioned native registry. Primitive
hardware events use declarative Atomic Schemas; composite operations such as
pulses and ramps use precompiled Morphism Templates in the shared arena; and
only language-shaping behavior such as replacement, identity, loop formation,
template binding, and unit conversion uses a closed Rust Special Form. This
avoids both Python generator execution and a handwritten lowering callback for
every standard-library operation. Registry semantic digests are ordinary Query
DAG inputs.

Compilation starts from the requested entry and lazily follows referenced
symbols. When a Source Module is first needed, the compiler registers its
imports, declarations, and stable `DefinitionId`s before analyzing bodies. A
static Symbol Resolver maps aliases, re-exports, constants, intrinsics, and
service/module instances to native identities. This permits recursive
definitions to be diagnosed without relying on Python object identity.

Every loaded module is parsed once into a NAC3 AST and receives a declaration-
only Module Index. Function, property, and constructor bodies are lowered to
Source HIR only when a resolved use makes their `DefinitionId` reachable.
Mutually recursive reachable definitions are constraint-generated and unified
as one strongly connected component. Consequently, unrelated host functions
need to be syntactically valid but need not satisfy the restricted CatSeq
language.

Compile-visible top-level bindings are Global Definitions with stable IDs and
lazy pure initializer HIR. A compiled read evaluates only that initializer and
records its actual global, intrinsic, class-schema, and environment dependencies.
Module-level Channels and service/module instances therefore become native IDs
without running the surrounding Python module. Host top-level expressions,
device setup, and `if __name__ == "__main__"` remain unevaluated unless a
reachable global incorrectly depends on them.

Only compile entries require an explicit source marker such as `@arena_build`.
Resolved service/module calls and property reads inherit Compile Reachability
transitively; they do not need duplicate decorators. Other methods in the same
class remain host definitions. If a reachable call targets host-only or
unsupported behavior, compilation fails at that call edge without importing
the class or falling back to Python execution.

An import edge establishes namespace visibility, but only a resolved use adds a
Definition DAG edge. Incremental invalidation therefore follows the definitions
that were actually referenced rather than every file that happened to import a
changed module. Module identity is the Source Bundle root plus canonical module
name; source and normalized-definition digests determine cache reuse.

Each external scan input receives a stable `RuntimeValueId`, called a Runtime
Slot. Expressions derived from Runtime Slots remain symbolic through source HIR,
template specialization, and relative RTMQ lowering. RTMQ linking receives a
separate Runtime Bindings map and evaluates those expressions into event offsets,
fragment durations, and supported numeric operation arguments. Runtime Slots
cannot determine a call target, board, channel, opcode, event count, or other
topology.

Scan containers are identified by their `ExpParams` annotation, not by a magic
parameter name. Parameters follow normal lexical shadowing rules over module
imports. Scan slots and call targets are collected only from HIR nodes reachable
from the function's returned root. A scan-derived callable or channel-dictionary
key is rejected because it would change DAG topology.

For `RydbergTransferExp.build_sequence`, this means:

- `params[self.pulse_time]` resolves to a runtime scan slot;
- `self.amp_420`, `self.amp_1013`, and `np.pi` resolve as compile-time values;
- service method calls resolve to stable template definitions;
- the existing `get_end_state` call and explicit state argument to `rx_BB1` are
  removed during rb1-next migration; Serial supplies that state implicitly;
- unrelated methods such as `prepare_run` are not compiled.

## Current vertical slice

The production path uses `nac3parser`/`nac3ast`, static bundle loading, and
definition-segmented Typed Source HIR. `catseqc check` returns a compact typed
summary, `catseqc emit-hir` is the explicit diagnostic projection, and
`catseqc emit-arena` directly evaluates the selected entry into the new
Python-free canonical Morphism and Value Expression arenas. `catseqc compile`
continues through native duration analysis, channel binding, RTMQ event
scheduling, board-local OASM wait insertion, and emits a versioned,
epoch-segmented `OASMCallPlan` JSON response.

The canonical arena is separate from the older compatibility arena. It stores
nodes in one append-only vector, children and Serial boundary policies in flat
range tables, and only native definition, operation, channel, payload, and
provenance IDs in nodes. Associative Serial and Parallel runs are flattened;
unreachable builder intermediates are removed before publication. Contextual
channel dictionaries become Parallel groups of channel-bound `Instantiate`
leaves. Resolved service calls become shared `DefinitionRef` leaves. No
`SourceCall`, `DeferredApply`, Python container, syntax node, or Source HIR
owner is present in the emitted arena.

For `RydbergTransferExp.build_sequence`, the current result contains 38
reachable arena nodes. Its root is one variadic Serial with 27 children; the
arena also contains one explicit Parallel group, three channel-bound
Instantiate leaves, three shared hardware-template bodies, waits, and 13
referenced service definitions. A composite `pulse >> hold` template remains
one shared template body behind one Instantiate leaf rather than being inlined
through its binding boundary. The full reachable type-analysis graph remains
47 definitions and 4,176 Typed Source HIR nodes.

The 0.3 target slice now specializes every reachable `DefinitionRef` before
target lowering and compiles the real `RydbergTransferExp.build_sequence`
entry end to end. Wait and Atomic payloads carry native `ValueExprId`s;
duration units are converted with the Target Profile clock using checked
exact-decimal arithmetic. Strict profiles require an integer Cycle Count while
profiles that explicitly select nearest-even quantization may round. Link-valued
scan expressions remain Runtime Slots until a separate Link Bindings document
is supplied.

The RTMQ backend resolves registered Atomic target schemas, calculates relative
Serial/Parallel/Loop timing without reconstructing lanes, tracks RWG waveform
and RF state plus RSP PID configuration, and emits complete TTL, RWG, RSP,
hardware-loop, global-sync, and opaque-host OASM calls. A global sync is a
structural epoch boundary: calls after it receive the next epoch even when the
sync has zero duration. Opaque regions use a Compile Environment binding with a
stable callable key and native JSON `args`/`kwargs`; Python objects never enter
the arena or plan. The Python adapter resolves those keys through an explicit
host registry and converts native record payloads into the existing OASM value
types.

Source and Morphism-arena durations describe only the experiment's logical
timeline. They do not include the time taken by OASM instructions such as RWG
waveform loads, carrier writes, play commits, or synchronization calls. The
target lowering owns the RTMQ ABI occupancy model, schedules those instructions
against each board-local cursor, and inserts waits only while producing the
OASM Call Plan. Hardware-loop occupancy corrections are likewise applied at
this boundary; they never mutate the canonical Morphism arena.

The current real-program acceptance plan has two epochs and 797 calls. It
contains native loop delimiters, 211 RWG waveform loads, sync calls on all ten
target boards, RSP PID state transitions, and two explicitly bound opaque
calibration calls. An external acceptance procedure assembles both compiler
paths with deterministic calibration bindings and requires the complete
disassembly of every generated board program to equal the pinned 0.2.4 result.
It also rejects leaked normalized-Python tokens and verifies that at least one
ramp contains a non-zero coefficient. This procedure remains outside the
standalone CatSeq test suite because its source program belongs to rb1-next.

This completes `OASMCallPlan` generation for the agreed 0.3 scope. Hardware
`if` and `SyncPhi` lowering remain intentionally outside that scope; an `if`
inside compiled source must resolve to a compile-time Boolean. Current
DefinitionRef IDs intern the fully resolved canonical call name; the final
bundle/module/name/kind `DefinitionKey`, bound `InstanceId`, and
per-child/per-boundary provenance tables remain pending and the current IDs
must not be persisted as final
artifact identity.

Release measurements over 100 unchanged-cache processes put the direct Typed
HIR-to-arena lowering at 31.8 microseconds mean and 35.3 microseconds p95. The
complete one-shot `emit-arena` process averages 8.3 ms warm (11.5 ms p95) and
71.8 ms cold; warm time is therefore dominated by process startup, decoding
the full HIR cache, and serializing the 19.8 KB debug response rather than DAG
construction.

For the full real-program `compile` path, a seven-run release sample on the
development host measured 16.8 ms median for native specialization plus OASM
planning. End-to-end process time was 90.2 ms without a frontend cache and
28.8 ms with an unchanged cache; the emitted JSON plan/report was 86.4 KB.
These figures deliberately separate the native compiler timer from process
startup, source discovery/cache work, and diagnostic JSON serialization.

## Typed Source HIR milestone

The accepted next implementation slice ends at definition-segmented Typed
Source HIR for the real `RydbergTransferExp.build_sequence` reachable graph:

```text
rydberg_transfer.py and reachable definitions
  -> nac3parser / nac3ast
  -> static imports, Module Index, and Compile Reachability
  -> per-definition Source HIR
  -> name resolution and recursive-SCC semantic analysis
  -> Typed Source HIR, Type Signatures, and diagnostics
```

This slice recursively analyzes reachable experiment, service, and module
methods; infers the confirmed CatSeq types; normalizes supported class
attributes, properties, dataclass replacement, and contextual aggregates; and
classifies scan values as typed Link inputs. It also implements the rustc-style
previous/current DepGraph, stable fingerprints, owner/projection query boundary,
selected result cache, and successful-session publication required to
demonstrate reuse across one-shot compiler invocations.

The native abstract-evaluation slice now consumes this Typed Source HIR without
introducing another frontend and emits the definition-linked Morphism and Value
Expression arenas described above. The first direct TTL path also reaches an
OASM Call Plan. Full definition specialization, State Environment validation,
physical Phase Frame behavior, reusable RTMQ fragments, and the complete Atomic
Schema registry remain outside the completed slice. The original HIR acceptance
plan is in `docs/dev/0.3_typed_source_hir_plan.md`.

The following work is intentionally still pending:

- replace the current recursive-SCC analysis facade with the complete
  constraint-generation and unification implementation;
- assign final bundle/module/name/kind Definition Keys and bound Instance IDs
  before DefinitionRef publication;
- specialize referenced definition bodies and evaluate remaining compile-time
  structural control flow;
- complete Value Expression lowering for compile fields, all registered scalar
  intrinsics, and operation arguments beyond duration;
- infer Morphism Effects and validate State Environments, Phase Frames, and
  per-child/per-boundary provenance;
- add the versioned Compile Request and publish the two native arenas as a
  relocatable artifact;
- generate reusable RTMQ fragment work products, complete operation
  relocations, Sync Phi epochs, and all registered OASM calls.

## Incremental boundary

`SourceCompilerSession` owns one arena and caches each `(source file, qualified
entry)` artifact. There are two reuse paths:

- identical source reuses parsing, source HIR, name resolution, and the arena
  template directly;
- changed module source is parsed and lowered again, but an identical source
  HIR structure plus identical resolved paths and scan slots reuses the existing
  arena root. Source spans are rebound to the new HIR and a lightweight template
  handle pins that revision; no arena nodes are rebuilt. This handles changes to
  unrelated host-side code without rebuilding the Morphism DAG.

Any changed name-resolution environment invalidates reuse even when the HIR
syntax itself is identical. A changed sequence HIR builds a new immutable
template segment; an already returned artifact remains valid because arena
storage is append-only.

This is currently definition-local incremental compilation. Once reachable
service calls resolve to `DefinitionId`, those definitions and call edges form
the dependency DAG: changing one definition invalidates its specialization and
transitive callers, while unrelated definitions remain cached. Source HIR is
therefore the frontend invalidation boundary, not the final dependency graph.
The definition graph and specialization cache carry invalidation across
functions.

The completed native frontend represents this as a demand-driven Query DAG.
Module parsing and indexing, definition lowering and resolution, SCC type
analysis, compile evaluation, and specialization are memoized queries that
record the exact earlier results and individual Compile Environment facts they
read, in read order. Each rustc-style Dep Node consists of a query kind and a
stable query-key fingerprint. Red-green validation compares normalized result
fingerprints: a source or environment input may change without propagating
further when the query's semantic output remains identical. The compiler does
not hash the entire Compile Environment into every cache entry.

The semantic query boundary separates `InferTypes(SccId)`,
`InferAvailability(SccId)`, and `AnalyzeDependencyRoles(SccId)`. Their stable
outputs are respectively Type Signatures, Availability Transfers, and
Structural Dependency Summaries stored in Semantic Fact side tables. A Link
value reaching a Structural consumer is diagnosed by availability analysis;
role analysis determines invalidation and relocation without inventing a new
base type.

At the end of a successful one-shot invocation, the new Query DAG, stable query
and result fingerprints, selected serialized query results, and reusable native
work products are published as one immutable incremental session. The next
invocation maps stable Definition Keys back to its dense session IDs and lazily
loads cached values. Failed compilation never replaces the last valid session.

This follows rustc's previous/current dep-graph model. The previous graph is
loaded as immutable flat node and ordered-edge arrays while the current graph is
built and streamed. `try_mark_green` locates an old Dep Node by stable key
fingerprint, validates dependencies in their original read order, and copies a
green node and its edges without executing its provider. A dirty provider is
rerun and may still become green when its stable result fingerprint is
unchanged.

Dep-graph metadata and fingerprints are distinct from the selected on-disk
query-result cache and from the Work Product index. Cache promotion retains
persisted results for green nodes that were not otherwise loaded. Module-level
aggregates expose per-definition Projection Queries as invalidation firewalls.
Canonical Programs, RTMQ Fragment Templates, and Relocatable Artifacts are Work
Products analogous to rustc code-generation outputs.

Query granularity follows semantic owners rather than individual expressions.
Input Dep Kinds expose module text and individual environment or target facts;
module queries parse and index; `DefinitionHeader(DefinitionKey)` projects one
declaration; definition-owner queries lower and resolve one Source HIR Segment;
SCC queries infer type, availability, and roles; specialization queries emit and
check canonical Morphism structure; and target queries lower one RTMQ Fragment
Template before an entry query assembles the Relocatable Artifact. Expression
facts remain dense side tables within their definition owner and never require
cross-session Dep Nodes.

`LinkArtifact(ArtifactKey, BindingDigest)` is batch-local and is not persisted by
default. The Relocatable Artifact and its Slot-to-fragment reverse index are the
cross-session reuse boundary for scan changes. If one child of `a >> b >> c`
changes duration, the unchanged child fragment Work Products remain green while
entry composition and relative offset linking rerun.

The implementation is CatSeq-owned rather than a dependency on rustc's private
query crates. Its initial encoding uses simple flat arrays; rustc-style packed
edge widths and parallel encoder allocation remain profile-driven physical
optimizations.

Query-key and result fingerprints are separate 128-bit values computed by a
CatSeq `StableHashingContext`. It maps Definition, Compile Instance, Source HIR,
Morphism, and Value Expression IDs to stable keys or structural child
fingerprints before hashing. Pointers, dense IDs, allocation and map iteration
order, comments, whitespace, and absolute spans never enter semantic hashes.
Semantic child order, typed values, normalized definition digests, Structural
bindings, and exactly read input facts do.

Raw module text remains an input fingerprint, so a whitespace edit reruns the
parser. If the normalized parser result is unchanged, its semantic result
fingerprint remains green and definition consumers do not run. Work Products
store Source Anchors formed from a Definition Key and owner-local node identity;
the current session maps anchors to current spans. Cache headers include the
compiler build and encoding version and reject an incompatible previous session
before any Dep Node is reused.

Query providers are deterministic over their key, declared input Dep Nodes, and
other query results. Files, environment state, clocks, randomness, live Python
objects, and mutable registries enter only through driver-fed input queries.
Providers return values and stable Source-Anchor-based Diagnostic Sets rather
than emitting side effects; `CollectDiagnostics(EntryKey)` aggregates the
reachable outcomes. CatSeq therefore does not initially implement rustc's
general Query Side Effect replay.

A provider may build directly in an engine-owned append-only arena segment, but
the segment is private until the query succeeds. Publication of the segment and
registration of any Work Product are atomic engine operations. Failed segments
are unreachable and never enter the current session. Cached-result decoding is
dependency-free and may not invoke providers or add Query DAG edges.

Runtime scan values never enter `compile_source`, the Compile Environment, or a
specialization key. Runtime Bindings populate `RuntimeValueId` slots during RTMQ
linking, so a scan update reuses the source HIR, arena template, and specialized
relative RTMQ fragment.

## Storage model

`ArenaStore` is the single logical store. Physical segments separate module
templates, service-bound templates, and experiment programs without requiring
node copies. Nodes use store-checked `(segment, local node)` references. Mutable
segments are append-only; published template segments are immutable.

Source lowering is iterative and memoized by `ExpressionId`. A source call is
stored as an unresolved `SourceCall` leaf; only Morphism composition structure
is copied into the arena. Local HIR sharing therefore becomes arena DAG sharing
instead of subtree expansion. Channel masks remain unresolved (`0`) until
service and hardware-map resolution. Published templates and frozen source
programs retain a type-erased Rust owner for that HIR, so copied handles cannot
outlive the payload interpretation they need.

The final canonical arena stores composition only as variadic Serial and
Parallel nodes. Child references occupy ranges in one flat edge table. Serial
also references one automatic-or-strict state policy per adjacent child
boundary. Typed HIR desugars `a >> {channel: transition}` into a Serial child
followed by a Parallel group of channel-bound template instantiation leaves;
there is no canonical ApplyMap or DeferredApply node. Associative runs are
flattened without crossing template instantiation or definition boundaries.

Template instantiation appends one `Instantiate` node. The compiler traverses
the referenced template with an explicit stack and an instance environment; it
does not expand the template into the program segment. A frozen program pins the
store and root. Later parents cannot change the graph reachable from that root.

`LoopNode` is a non-composition control node with one body edge, a trip-count
Value Expression, and a Morphism Effect. Compile-reachable Python sequencing
`for` and compatibility `repeat_morphism` calls lower to this node without
copying their bodies. A target either lowers supported induction and loop-
carried expressions to a native hardware loop or reports a target capability
error; it does not silently unroll the Morphism DAG.

`SyncPhiNode` is a non-composition control leaf. All participating boards arrive
at its rendezvous before the coordinator releases the next Epoch. The following
Serial child is scheduled from offset zero of that runtime release origin rather
than from an absolute timestamp derived from the preceding region.

## Required invariants

- A node can only refer to nodes in the same logical store.
- Template segments cannot depend on program segments.
- Publishing a template freezes its segment.
- Composition never mutates either input root.
- Freezing never copies or consolidates reachable nodes.
- Compiler traversal is iterative and carries template instance context.
- Python objects and Python callables do not become native DAG payloads.
- Service and module `InstanceId`s are immutable; coherent logical phase uses
  registered Phase Frames and Morphism Effects rather than mutable trackers.
- Morphism arena payloads do not refer back to Python AST or Source HIR;
  symbolic operands use native `ValueExprId`s.
- Published artifacts do not own or pin Source HIR; compiler-session caches may
  retain it independently for incremental compilation and diagnostics.
- Typed Source HIR Contextual Aggregates are eliminated before Morphism arena
  lowering; the arena contains no list, tuple, dictionary, or optional nodes.
- Native source does not expose State Environments as `StateMap` values or
  permit `get_end_state`; Serial edges carry state dependencies.
- Canonical composition nodes are variadic Serial or Parallel; dictionary
  application does not introduce a third composition kind.
- Sequencing loops retain one canonical Loop node and body edge. Their iteration
  boundary effects must compose legally, and target lowering never silently
  expands them into repeated Serial children.
- A Sync Phi ends one Epoch and establishes the runtime origin of the next; all
  participating boards must take part in its rendezvous.
- All incoming paths to a Sync Phi have identical externally visible end states
  for every affected channel. State recovery occurs before the rendezvous;
  Sync Phi never constructs a state union.
- Time comparison, subtraction, and scheduling optimization cannot cross a Sync
  Phi boundary as if both sides shared one absolute origin.
- A Dynamic Timing Contract cannot have an ordinarily scheduled successor in
  the same Epoch; reconvergence through Sync Phi is mandatory.
- Duration conversion never narrows or rounds to a Cycle Count implicitly.
- Parallel composition has one shared end boundary. Its duration is the
  symbolic maximum of all child durations, and every shorter child holds its
  final state until that boundary before a following serial child may start.
- Parallel Alignment is semantic and padding is not stored as explicit nodes in
  the canonical arena. Compatibility Lane views and target lowering materialize
  equivalent identity waits wherever their output formats require them.

## Compilation artifacts

The compiler caches a target-independent Canonical Program separately from its
target-specific Relocatable RTMQ Artifact. The Canonical Program owns the native
Morphism and Value Expression arenas, completed Morphism Effects, schemas, and
provenance. Target lowering validates effects, selects Atomic implementations,
partitions boards, and lowers Loop and Sync Phi control into a relative RTMQ
fragment DAG. Relative offsets and operands remain Value Expressions.

Module templates retain channel and parameter slots. A service artifact binds
the structural portions of hardware maps, calibration snapshots, and instance
parameters while keeping a relative event fragment. Topology-independent
external scalars remain Environment Slots. Link Bindings supply those slots and
Runtime Slots, then perform incremental expression evaluation, relocation, and
board linking. The linked OASM Call Plan keeps calls relative to their Epoch
origin; the initial origin may be known before runtime, whereas a post-Sync-Phi
origin is established by the hardware rendezvous.

The relocatable artifact stores reverse indexes from Link Slots through Value
Expressions to affected board/Epoch fragments. Linking therefore does not rerun
source analysis, specialization, state validation, or instruction selection. A
Target Profile change rebuilds target fragments while retaining a compatible
Canonical Program.

Relative RTMQ storage remains a segmented fragment DAG until OASM Call Plan
emission. A target-lowered definition specialization owns a Fragment Template;
call sites create Instantiate nodes with channel and value bindings instead of
copying events. Variadic Serial and Parallel, Loop, Sync Phi, and Epoch nodes
retain their structure. Associative edges may flatten within one specialization
segment but not across a definition or control boundary. Link recomputes only
affected fragment summaries and their ancestors before the final board/Epoch
traversal produces ordered calls.

Stable `DefinitionId` values index an append-only definition table. A template
identity begins as a deterministic `DefinitionKey` containing its Source Bundle,
canonical module, qualified name, and kind; the session interns it as a dense
integer ID. Source edits preserve that identity but create a new immutable
`DefinitionRevision` with separate interface and implementation digests.

A template Specialization Key contains the definition revision, structural
parameter digest, structural instance and hardware bindings, structurally used
calibrations, intrinsic schema versions, and dependency revisions. It excludes
Runtime and Environment Slot values. Changing a scan point or a nonstructural
calibration therefore evaluates Link Values and relinks the same RTMQ artifact
instead of reparsing source or recompiling the template. Persistent caches use
deterministic keys and digests rather than session-local integer IDs.

Base types and Value Availability do not form additional specialization axes.
The Definition Revision's interface identity already covers its single Type
Signature, while its Availability Transfer permits symbolic Value Expression
parameters to be bound to constants or Link Slots without cloning the source
definition. A helper called with incompatible base types is rejected instead of
being implicitly monomorphized.

One compile-and-link invocation may accept a batch of Link Bindings and emit one
OASM Call Plan per scan point from the same relocatable artifact. Adaptive scans
may call a lightweight artifact linker that does not initialize the source
frontend. Neither path requires a persistent compiler process.

`catseqc compile`, `catseqc link`, and `catseqc check` are CLI adapters over this
same Rust request interface. A request without Link Bindings emits only the
relocatable artifact; a request with bindings compiles once and links every
provided point. Structured incremental statistics report green and red queries,
reused work products, and definitions or target fragments rebuilt.

The Python DAG compiler remains the correctness oracle during 0.3 development.
`catseqc` becomes the default only after differential Rydberg checks match
events, calls, and integer assembly. On 2026-07-15, the external
`RydbergTransferExp.build_sequence` acceptance procedure matched the exact
0.2.4 assembly on all ten emitted board programs. Other production sequences
must receive the same coverage before the default changes.
