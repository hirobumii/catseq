# CatSeq 0.3 native compiler

Status: authoritative current implementation status, verified 2026-07-16

CatSeq 0.3 is a Python 3.12 restricted-source language with a Rust-owned native
compiler. The source spelling of `Morphism`, `MorphismDef`, `>>`, `@`, `|`, and
channel dictionaries remains Python, but production compilation does not import
or execute the experiment module.

This document is the single status summary for the 0.3 compiler. ADRs record
accepted decisions; milestone plans record implementation history. If either
conflicts with this document about what is currently selected in production,
this document is authoritative.

## Supported application seam

The supported application API is:

```python
from catseq.compilation import compile_entry, execute_oasm_calls

result = compile_entry(
    experiment.build_sequence,
    params,
    environment=environment,
)
calls = result.to_oasm_calls(opaque_callables=opaque_callables)
_, exp_sequence = execute_oasm_calls(calls, assembler_seq)
```

`compile_entry()` uses the bound function to locate its source definition and
build restricted compiler bindings. It does not call the function. JSON-safe
scalar and sequence values from arguments and the owning object are encoded as
candidate runtime bindings; live objects and non-encodable host state do not
enter the request. It returns an `OASMCompileResult`, including the versioned
plan, target-clock metadata, and `logical_duration_cycles` needed by host
execution-timeout contracts.

The default route is in process:

```text
compile_entry()
  -> schema-versioned, Python-free Compile Request
  -> catseq._native.compile(bytes)
  -> catseq_compiler::compile_json_request(&[u8])
  -> static frontend and native compiler passes
  -> schema-versioned OASMCallPlan response
  -> OASMCompileResult
  -> host-owned Python OASM assembler
```

Passing `compiler=` to `compile_entry()` or setting `CATSEQC_BIN` selects an
external `catseqc compile` process explicitly. This compatibility and diagnostic
route sends the same logical request to the same Rust compiler implementation;
it is not a second production compiler.

## Ownership boundaries

The implementation is divided as follows:

```text
catseq/compilation       public facade, result decoding, OASM adapter
rust/catseq-frontend     source loading, NAC3 parsing, Source HIR, analysis,
                         diagnostics, incremental query session
rust/catseq-core         shared compiler types and canonical arenas
rust/catseq-compiler     shared request-to-response compiler entry and CLI logic
rust/catseq-rtmq         RTMQ specialization, scheduling, and OASMCallPlan
rust/catseq-python       thin PyO3 byte-request adapter
rust/catseqc             thin standalone executable adapter
```

The Rust compiler owns source analysis, reachability, type and availability
facts, canonical Morphism and Value Expression arenas, duration and state
semantics, RTMQ instruction occupancy, scheduling, and plan generation. The
Python OASM adapter only resolves explicitly registered opaque host callables
and converts plan records into calls on the consumer-owned assembler.

Consumer repositories own their source bundle, compile environment, hardware
map, scan/link bindings, opaque callable registry, assembler, and end-to-end
differential acceptance checks. CatSeq does not fetch, import, or depend on a
consumer repository during build, test, packaging, or release.

## Current source and compiler path

The selected frontend uses revision-pinned `nac3parser` and `nac3ast` crates.
It statically loads the source bundle, resolves compile-reachable imports and
definitions, and normalizes the supported Python subset into CatSeq-owned,
definition-segmented Source HIR. It never imports application modules through
CPython.

The active path is:

```text
Source Bundle and Compile Environment
  -> static imports and compile reachability
  -> definition-segmented Typed Source HIR
  -> canonical Value Expression and Morphism arenas
  -> definition specialization and target lowering
  -> epoch-segmented, board-grouped OASMCallPlan
```

Supported source constructs include registered compiler declarations,
restricted class/dataclass projections, static properties, compile-time
structural conditionals, supported sequencing loops, scan values that do not
select topology, channel dictionaries, native hardware intrinsics, global sync,
and explicitly registered opaque host calls. Unsupported reachable Python
behavior is a source-anchored compiler diagnostic; unreachable host lifecycle
code is not compiled.

The compiler uses a rustc-style one-shot incremental session. Requests load the
previous immutable dependency graph and selected cached results, validate
red/green queries, and atomically publish a successful replacement. A resident
compiler process is not required for cache reuse.

## Commands and artifacts

The packaged and standalone `catseqc` adapters expose exactly four commands:

```text
catseqc check
catseqc emit-hir
catseqc emit-arena
catseqc compile
```

- `check` returns diagnostics and a compact typed summary.
- `emit-hir` emits the explicit Source HIR diagnostic projection.
- `emit-arena` emits the Python-free canonical arena projection.
- `compile` continues through RTMQ lowering and emits an `OASMCallPlan`.

There is no independent `catseqc link` command in 0.3. Link bindings are part
of the compile request. The application-facing Python facade remains the stable
seam; CLI output is primarily for diagnostics, CI, compiler development, and
standalone automation.

The final Rust artifact is an `OASMCallPlan`, not an RTMQ binary. Direct binary
generation remains outside the 0.3 boundary; encoding is delegated to the
host-owned OASM assembler without moving compiler semantics into Python.

## Packaging and versions

Platform wheels contain the Python package and one PyO3 extension. The
`catseqc` console entry point calls the CLI embedded in that extension, avoiding
a duplicate native compiler in the wheel. Standalone `catseqc` archives remain
available for non-Python automation.

Python project metadata follows PEP 440 and is currently `0.3.1.dev0`. Cargo
metadata follows SemVer and is currently `0.3.1-dev.0`. These are
ecosystem-native spellings of the same development prerelease.

## Retained cleanup boundaries

- The older tree-sitter frontend source and dependencies are still present in
  the workspace but are not selected by `check`, `emit-hir`, `emit-arena`, or
  `compile`. Removing that code is a separate cleanup change so dependency and
  consumer gates can isolate regressions.
- The Python OASM adapter remains an intentional execution boundary. Tightening
  its schema or reducing conversion work must preserve the versioned
  `OASMCallPlan` and consumer acceptance contract.
- Historical performance measurements and milestone-specific acceptance data
  remain in the archived
  [`0.3_typed_source_hir_plan.md`](0.3_typed_source_hir_plan.md); they are not
  claims about the current development checkout.

## Verification contract

Every implementation change runs Python lint and tests plus the Rust workspace
format, Clippy, and test gates. Packaging or adapter changes additionally
rebuild the wheel from an sdist and smoke-test the installed wheel by calling
`catseq._native.compile()`. Consumer repositories run their own production
source acceptance gate against the CatSeq checkout or installed artifact.

For the current development checkout, also verify both ecosystem versions:

```bash
uv run python -c "import importlib.metadata as m; print(m.version('catseq'))"
rust/target/debug/catseqc --version
```

The expected outputs are `0.3.1.dev0` and `catseqc 0.3.1-dev.0`.
