# CatSeq 0.3.1 Rydberg-transfer pipeline performance

Measured on 2026-07-23 with Python 3.12.11 and a release wheel rebuilt from
the staged CatSeq source distribution. The CatSeq working tree was based on
`ef32ee61f0547063796f49ef10968432b6441c18`; the RB1 source workload was based
on `ce878065fdaf5235315686907728ce423925bd83`. Both repositories were recorded
as dirty because this checkpoint was measured before its implementation commit
and the RB1 checkout contained local work.

  /home/hermes/workspaces/exp/catseq/benchmarks/rydberg_transfer_pipeline.py \

The imported release extension was 7,262,136 bytes with SHA-256
`15d07ef549ccd01c2c7cbe4cb399abf5e18c85124af2c61b36dc5db1845570c5`,
built with `rustc 1.95.0 (59807616e 2026-04-14)`. The sdist SHA-256 was
`1248b615ac44bfeff7fb2368103e5cb732455f9c250e7e1687863c2e77e3d8bf`;
the rebuilt wheel SHA-256 was
`c12e6aef5577299f9f331d45b65411d586842784e549e77ab04445bc7d3125fc`.
The benchmark deliberately uses the interface name
`catseq-benchmark-no-device`, which exceeds Linux's 15-byte interface-name
limit and therefore cannot identify a real interface. It validates the Rust
runtime contract and materializes the Download loader, then stops at
`transport_open_failed` before opening a raw socket. It sends no hardware
packets.

| Stage | Median (ms) | p95 (ms) |
| --- | ---: | ---: |
| Build compile request | 43.980 | 44.426 |
| Encode request JSON | 0.033 | 0.035 |
| Native compile, cold cache | 110.204 | 116.604 |
| Native compile, warm cache | 22.875 | 28.731 |
| Decode compiler response JSON | 0.386 | 0.400 |
| Decode OASM Call Plan | 0.820 | 0.871 |
| Create OASM assembler contexts | 0.030 | 0.033 |
| OASM assembly and completion finalization | 15.950 | 16.360 |
| Rust validation, loader materialization, safe open failure | 0.088 | 0.093 |
| Hot offline pipeline total | 40.105 | 45.522 |

The hot total starts with an already-built JSON request and includes warm
native compilation, response decoding, Call Plan decoding, OASM assembly,
Rust validation, loader materialization, and the deliberate transport-open
failure. It does not include compile-request construction or request JSON
encoding.

Cold native compilation dominates the first run. On a warm cache, native
compilation and OASM assembly are the two material costs. JSON transport, plan
decoding, assembler setup, and Rust preparation are all below 1 ms at their
medians.

An earlier run from the source checkout loaded the 75 MB debug PyO3 extension
and measured 1208.489 ms cold and 142.653 ms warm. Those values were rejected
as deployment-performance evidence and replaced by the release-wheel results
above.

## Live Rust download timing

The live acceptance case is the reviewed two-board `nop(2)` program, not the
Rydberg experiment. It uses:

- interface `eno1`;
- source MAC `60:cf:84:a7:bb:ff`;
- chassis-2 destination MAC `60:cf:84:a7:bc:01` (host MAC + 2);
- reply node 21/channel 0;
- `rwg0 -> node 2/channel 0`;
- `rwg1 -> node 5/channel 0`.

`execute_oasm_program` was measured around the native call only. The interval
includes Rust loader materialization, AF_PACKET open/send, completion
monitoring, and close. It excludes Python process startup, pinned-OASM fixture
verification, and construction of the Python handoff values.

Across 11 consecutive runs:

| Result | Value |
| --- | ---: |
| Successful runs | 11/11 |
| Median | 21.332 ms |
| p95 | 31.004 ms |
| Minimum | 14.331 ms |
| Maximum | 31.004 ms |

Every run reported both `rwg0` and `rwg1` as `succeeded`, with no device
exceptions.

## Why the full Rydberg plan was not downloaded

Chassis 2 currently contains only the two RWGs at nodes 2 and 5. The complete
Rydberg plan addresses ten boards, including a Master, one RSP, and RWGs that
are not installed in this chassis. Those targets would time out, and executing
only a physical subset would no longer be the measured Rydberg program.
Therefore the full workload is benchmarked through loader materialization
offline, while live download timing uses the explicitly reviewed two-board
no-op.

## Raw-packet permission

The successful command was:

```bash
uv run python tools/validate_chassis2_noop_download.py
```

No `sudo` was used. The uv-managed Python interpreter already has effective
raw-packet capability on the host. A managed sandbox can strip that capability,
so hardware validation must execute in the authorized host context.

For a durable least-privilege deployment, grant `CAP_NET_RAW` to a dedicated
runtime executable rather than a general-purpose Python interpreter. The
current successful setup proves that root privileges are unnecessary; it does
not make a broadly capability-enabled interpreter the preferred production
arrangement.
